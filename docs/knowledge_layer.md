# Provenance & Knowledge Layer

The observatory has a 3-layer knowledge system that turns individual project results into a connected, searchable research memory. Each layer serves a different audience and question type.

## Why This Exists

With 41 projects, the observatory generates hundreds of findings, figures, and data artifacts. Without structured indexing, answering questions like "which organisms have fitness data but no metal experiments?" requires reading all 41 project reports (~200K+ tokens). The knowledge layer makes cross-project discovery instant and enables the AI co-scientist to reason about research gaps, suggest next steps, and maintain continuity across sessions.

## The Three Layers

```
Layer 1 (per-project)    projects/<id>/provenance.yaml    "What did project X produce?"
Layer 2 (global index)   docs/project_registry.yaml       "Which projects studied topic Y?"
                         docs/figure_catalog.yaml
                         docs/findings_digest.md
Layer 3 (semantic graph) knowledge/                       "How do findings connect across projects?"
                         docs/knowledge_graph_coverage.md
                         docs/knowledge_gaps.md
```

### Layer 1: Structured Provenance

Each project gets a `provenance.yaml` sidecar file containing machine-readable metadata extracted from its report. This includes:

- **References** with DOI/PMID, classified by role (primary data source, supporting, contradicting, methodology)
- **Data sources** linking to BERDL collection IDs and specific tables
- **Cross-project dependencies** (which projects feed into or extend this one)
- **Findings** linked to specific notebooks, figures, and statistical evidence
- **Generated data** with row counts and source notebooks

Provenance files are generated by `tools/generate_provenance.py`, which reads project markdown and uses the Claude API (Haiku by default) to extract structured metadata. They can also be written manually or by `/synthesize`.

### Layer 2: Knowledge Registry

A set of global index files derived from all provenance.yaml files (with markdown fallback for projects without provenance). Generated by `scripts/build_registry.py`.

| File | Contents | Size |
|------|----------|------|
| `docs/project_registry.yaml` | All projects with status, questions, findings, tags, deps | ~20K tokens |
| `docs/figure_catalog.yaml` | 330 figures with captions, tags, project links | ~15K tokens |
| `docs/findings_digest.md` | ~195 findings as a concise searchable summary | ~8K tokens |

The registry is always re-derivable from project files — if it drifts, `/build-registry` regenerates it completely. Skills treat it as a read cache, not the source of truth.

### Layer 3: Semantic Knowledge Graph

The knowledge graph captures biological entities, their relationships, the hypothesis lifecycle, and a research timeline across all projects. It lives in `knowledge/` and follows a LinkML schema (`knowledge/schema/research_graph.yaml`).

| File | What it contains |
|------|-----------------|
| `knowledge/entities/organisms.yaml` | 20 organisms with NCBI taxonomy IDs |
| `knowledge/entities/genes.yaml` | 15 genes/gene groups with KEGG KO IDs |
| `knowledge/entities/pathways.yaml` | 10 metabolic pathways with KEGG IDs |
| `knowledge/entities/methods.yaml` | 14 analytical methods |
| `knowledge/entities/concepts.yaml` | 21 scientific concepts |
| `knowledge/relations.yaml` | 32 evidence-backed edges between entities |
| `knowledge/hypotheses.yaml` | 20 hypotheses tracking the evolution of understanding |
| `knowledge/timeline.yaml` | 43 chronological research events |

Two derived views are generated by `build_registry.py` into `docs/`:
- `knowledge_graph_coverage.md` — which entities and projects have graph representation
- `knowledge_gaps.md` — deterministic gap analysis (sparse organisms, method coverage gaps, untested hypotheses, unexplored entity pairs, co-mentioned pairs missing relation edges)

## How the Layers Connect

```
/synthesize completes a project
    |
    v
REPORT.md ──> provenance.yaml ──> build_registry.py ──> registry artifacts (Layer 2)
                                        |                       |
                                        v                       v
                               knowledge/ updates (Layer 3)   /knowledge queries
                                        |                   /suggest-research
                                        v
                               coverage + gap reports
```

Every project completion enriches all three layers. The `/synthesize` skill now **requires** Layer 3 updates (entities, relations, timeline, hypotheses) with a completion checklist before finishing.

## Skills That Use the Knowledge Layer

### Querying

| Skill | What it does | Layers used |
|-------|-------------|-------------|
| `/knowledge <topic>` | Search projects and findings by keyword | 2 |
| `/knowledge figures <topic>` | Search figure catalog | 2 |
| `/knowledge data <topic>` | Search reusable data artifacts | 2 |

### Research Workflow

| Skill | How it uses the knowledge layer |
|-------|-------------------------------|
| `/status` | Reads registry for in-progress projects, hypotheses for attention items, timeline for recent events, gaps for top research opportunities |
| `/suggest-research` | Reads registry for landscape analysis, runs `query_knowledge.py gaps` for entity-level gap detection, checks hypotheses for untested claims |
| `/interpret` | Checks hypotheses.yaml for related hypotheses from other projects to contextualize intermediate results |
| `/compare <A> <B>` | Reads registry entries and entity connections to produce side-by-side project or organism comparisons |
| `/synthesize` | **Writes** provenance.yaml (Layer 1), runs build_registry.py (Layer 2), updates entities/relations/hypotheses/timeline (Layer 3) |
| `/submit` | Validates provenance.yaml, runs build_registry.py |
| `/build-registry` | Regenerates all Layer 2 artifacts and Layer 3 coverage/gap reports |
| `/berdl_start` | Registers new projects in the registry immediately; Path 4 uses phase detection from registry data |

### Workflow Map

```
New project:     /status -> /suggest-research -> /berdl_start Path 1
                    -> /literature-review -> [analysis] -> /interpret
                    -> /synthesize -> /submit

Quick lookup:    /knowledge <topic>

Resume work:     /status -> /berdl_start Path 4

Cross-project:   /knowledge gaps -> /compare <A> <B> -> /suggest-research

After completion: /synthesize -> /submit -> /build-registry
```

## Scripts and Tools

### Generate provenance (Layer 1)

```bash
# Single project
uv run tools/generate_provenance.py <project_id>

# All projects missing provenance.yaml
uv run tools/generate_provenance.py

# Use a different model (default: claude-haiku-4-5-20251001)
uv run tools/generate_provenance.py --model claude-sonnet-4-6

# Preview without API calls
uv run tools/generate_provenance.py --dry-run
```

Requires `ANTHROPIC_API_KEY` environment variable.

### Rebuild registry (Layer 2 + Layer 3 reports)

```bash
# Full rebuild
uv run scripts/build_registry.py

# Single project update
uv run scripts/build_registry.py --project <project_id>

# Preview without writing
uv run scripts/build_registry.py --dry-run
```

### Query knowledge deterministically

```bash
uv run scripts/query_knowledge.py search "metal stress"
uv run scripts/query_knowledge.py landscape
uv run scripts/query_knowledge.py gaps
uv run scripts/query_knowledge.py backfill          # list projects missing Layer 3
uv run scripts/query_knowledge.py hypotheses testing
uv run scripts/query_knowledge.py connections org_adp1
uv run scripts/query_knowledge.py timeline
```

### Validate consistency

```bash
# Check provenance.yaml files against schema
uv run scripts/validate_provenance.py

# Check if registry artifacts are stale
uv run scripts/validate_registry_freshness.py

# Check semantic graph integrity
uv run scripts/validate_knowledge_graph.py
```

## Typical Workflows

### After completing a project

1. `/synthesize` writes REPORT.md, generates provenance.yaml, updates registry, updates knowledge graph
2. `/submit` validates and generates REVIEW.md
3. Address review feedback
4. Commit

All three layers are updated automatically by `/synthesize`.

### Backfilling older projects

24 projects were completed before Layer 3 existed and lack graph coverage.

```bash
# See which projects need backfilling
uv run scripts/query_knowledge.py backfill

# Backfill interactively via the skill
/knowledge backfill <project_id>
```

The backfill skill reads the project's REPORT.md and provenance.yaml, extracts entities/relations/hypotheses/timeline following the same logic as `/synthesize` Step 7.7, and presents proposed additions for confirmation before writing.

### When the registry feels stale

```bash
# Check freshness
uv run scripts/validate_registry_freshness.py

# Rebuild if needed
uv run scripts/build_registry.py
```

The `/knowledge` and `/suggest-research` skills automatically run the freshness check and warn if the registry is out of date.

### Starting a new session

Use `/status` to see:
- In-progress projects and their current phase
- Hypotheses needing attention (testing/proposed)
- Top research gaps
- Recent timeline events

This orients the co-scientist to pick up where the last session left off.

## Design Decisions

- **Structured YAML over vector databases**: At 41 projects, grep/glob on structured text outperforms the overhead of embedding infrastructure. Claude Code's native tools work directly on YAML. Revisit at 500+ projects.
- **Deterministic scripts over LLM queries**: `query_knowledge.py` and `build_registry.py` are pure Python with no LLM calls. Only `generate_provenance.py` uses the API (for initial extraction). This ensures reproducible, fast queries.
- **Layer 3 is required, not optional**: The knowledge graph was sparse (32 relations for 80 entities) because updates were optional. Making them mandatory in `/synthesize` ensures density grows with each project.
- **Freshness checks are advisory**: Skills warn about stale data but proceed regardless. Stale data is better than no data; blocking on staleness would break the workflow.
